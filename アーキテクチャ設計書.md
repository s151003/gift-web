# アーキテクチャ設計書
## ギフト券取引価格履歴確認システム

**バージョン:** 0.2
**作成日:** 2026-03-14

---

## 1. 概要

### 1.1 システムの目的

Amazonギフト券の二次流通市場における取引価格を、複数の取引所をまたいで継続的に収集・可視化するシステム。各取引所の現在の出品価格が、過去の相場に対して割安か割高かをユーザーが即座に判断できることを目的とする。

### 1.2 解決する問題

| 問題 | 解決策 |
|------|--------|
| 取引相場が毎時変動しており、現在価格の相対的な良し悪しが判断できない | 過去の相場推移をグラフ表示し、現在価格を文脈の中で捉えられるようにする |
| 複数の取引所を手動で巡回する必要がある | 4取引所のデータを1画面で横断比較できるようにする |

### 1.3 対象取引所

| ID | サイト名 | URL | 取引履歴 |
|----|---------|-----|---------|
| `ama-gift` | アマギフ | https://ama-gift.com/ | 未確認 |
| `giftissue` | ギフトイシュー | https://giftissue.com/ | 未確認 |
| `beterugift` | べてるギフト | https://beterugift.jp/ | 未確認（JSONエンドポイントの可能性あり） |
| `amaten` | アマテン | https://amaten.com/ | 未確認 |

> **NOTE:** 各サイトの取引履歴表示有無は実装前のスクレイピング調査で確認する。取引履歴が取得可能なサイトのみ `transactions` テーブルに保存する。

## 3. データ設計

### 3.1 price_snapshots（集計スナップショット）

1時間ごと × 1サイトごとに1レコード。スクレイプ時点の全出品を集計した値を保存する。

```sql
CREATE TABLE price_snapshots (
  id                   INTEGER PRIMARY KEY AUTOINCREMENT,
  site_name            TEXT    NOT NULL,   -- 'ama-gift' | 'giftissue' | 'beterugift' | 'amaten'
  scraped_at           TEXT    NOT NULL,   -- ISO 8601 UTC 例: '2026-03-14T10:00:00Z'
  listing_count        INTEGER NOT NULL,   -- 全出品数（生データ）
  best_discount_rate   REAL    NOT NULL,   -- 最大割引率（%）全件対象
  worst_discount_rate  REAL    NOT NULL,   -- 最小割引率（%）全件対象
  avg_discount_rate    REAL    NOT NULL,   -- 平均割引率（%）全件対象
  median_discount_rate REAL    NOT NULL,   -- 中央値（%）全件対象
  created_at           TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  UNIQUE(site_name, scraped_at)            -- 同一サイト・同一時刻の重複挿入を防ぐ
);

CREATE INDEX idx_snapshots_site_time ON price_snapshots(site_name, scraped_at DESC);
CREATE INDEX idx_snapshots_time      ON price_snapshots(scraped_at DESC);
```

**設計メモ:**
- 割引率は `(1 - listing_price / face_value) * 100` で計算
- `UNIQUE(site_name, scraped_at)` により、スクレイパーのリトライ時や二重実行時でも重複レコードが生じない。ingest 側は `INSERT OR REPLACE` を使用する
- `created_at` は `strftime('%Y-%m-%dT%H:%M:%SZ', 'now')` で UTC の ISO 8601 形式を保証（`datetime('now')` はタイムゾーン表記が付かないため統一）
- フロントエンド側でフィルタリング（例：`best_discount_rate` が極端な場合を除外）を行う設計とし、生の集計値を保存する
- 出品数（`listing_count`）を保存することで「出品が多い時間帯に割引率が高くなる」などの傾向分析が可能

### 3.2 transactions（取引履歴）

取引履歴を公開しているサイトのみ対象。同一取引の重複保存を避けるため、`site_name + traded_at + face_value + traded_price` の組み合わせでユニーク制約を設ける。

```sql
CREATE TABLE transactions (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  site_name      TEXT    NOT NULL,
  traded_at      TEXT    NOT NULL,   -- 取引成立日時（サイト表示値、UTC変換済み）
  face_value     INTEGER NOT NULL,   -- 額面（円）
  traded_price   INTEGER NOT NULL,   -- 実際の取引価格（円）
  discount_rate  REAL    NOT NULL,   -- (1 - traded_price / face_value) * 100
  scraped_at     TEXT    NOT NULL,   -- このレコードを取得した日時（UTC ISO 8601）
  created_at     TEXT    NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
  UNIQUE(site_name, traded_at, face_value, traded_price)
);

CREATE INDEX idx_transactions_site_time ON transactions(site_name, traded_at DESC);
```

**設計メモ:**
- `traded_at` はスクレイパー側でUTCに変換してから保存する（サイトが JST 表示の場合は `-9:00` して送信）
- スクレイプのたびに同じ履歴が再取得されるが、`UNIQUE` 制約により `INSERT OR IGNORE` で重複を回避
- `discount_rate` は冗長だがクエリ効率のため保持する

## 4. API設計

### 4.1 認証

GitHub Actions → Workers 間の書き込みAPIはBearerトークンで保護する。トークンはGitHub ActionsのSecretsとCloudflare Workers環境変数の両方に設定する。

```
Authorization: Bearer {INGEST_SECRET_TOKEN}
```

読み取りAPI（フロントエンド向け）は認証不要（公開データのため）。

**CORS:** フロントエンド（Cloudflare Pages）のオリジンからのリクエストを許可する。Workers 側で `Access-Control-Allow-Origin` を設定する。

### 4.2 入力バリデーション（ingest）

Workers 側で以下を検証し、不正なデータはDB手前で弾く：

| フィールド | バリデーション |
|---|---|
| `scraped_at` | ISO 8601 UTC 形式、現在時刻 ±1時間以内 |
| `site_name` | 許可リスト（`ama-gift` / `giftissue` / `beterugift` / `amaten`）のみ |
| `discount_rate` 各種 | 0 以上 100 未満、`best >= avg >= median >= worst` |
| `listing_count` | 0以上の整数 |
| `face_value` | 正の整数 |
| `traded_price` | 正の整数、`face_value` 未満 |

### 4.3 エンドポイント一覧

#### POST /api/ingest
スクレイパーからのデータ受け取り。

**リクエストボディ:**
```json
{
  "scraped_at": "2026-03-14T10:00:00Z",
  "snapshots": [
    {
      "site_name": "ama-gift",
      "listing_count": 142,
      "best_discount_rate": 9.5,
      "worst_discount_rate": 2.0,
      "avg_discount_rate": 6.3,
      "median_discount_rate": 6.1
    }
  ],
  "transactions": [
    {
      "site_name": "amaten",
      "traded_at": "2026-03-14T09:47:00Z",
      "face_value": 10000,
      "traded_price": 9200,
      "discount_rate": 8.0
    }
  ]
}
```

> **部分失敗の扱い:** スクレイパーが一部サイトの取得に失敗した場合、成功したサイトのみ `snapshots` に含めて送信する。`snapshots` が空配列でも `200 OK` を返す（送信自体は成功）。

**レスポンス:**
- `200 OK` — `{ "inserted_snapshots": 3, "inserted_transactions": 12 }`
- `400 Bad Request` — バリデーションエラー（詳細メッセージ付き）
- `401 Unauthorized` — 認証失敗
- `500 Internal Server Error` — DB書き込み失敗

#### GET /api/snapshots/latest
各取引所の最新スナップショット1件ずつを返す。サマリーカード表示用。

**レスポンス例:**
```json
{
  "data": [
    {
      "site_name": "ama-gift",
      "scraped_at": "2026-03-14T10:00:00Z",
      "listing_count": 142,
      "best_discount_rate": 9.5,
      "worst_discount_rate": 2.0,
      "avg_discount_rate": 6.3,
      "median_discount_rate": 6.1
    }
  ]
}
```

**実装クエリ（参考）:**
```sql
SELECT * FROM price_snapshots
WHERE (site_name, scraped_at) IN (
  SELECT site_name, MAX(scraped_at) FROM price_snapshots GROUP BY site_name
);
```

### 5.3 グラフ仕様

- **チャートタイプ:** line（ApexCharts）
- **X軸:** 時刻（scraped_at）
- **Y軸:** 割引率（%）
- **系列:** 各取引所を別系列で表示（最大4本）
- **ズーム:** ApexCharts 標準のブラシ/ズーム機能を有効化
- **ツールチップ:** ホバー時に全取引所の値を一覧表示（クロスヘア表示）
- **表示切替:** 最高割引率 / 平均割引率 / 出品数 をラジオボタンで切替

### 5.4 データ取得戦略

```
- GET /api/snapshots/latest  → サマリーカード（SWR, revalidateOnFocus: true）
- GET /api/snapshots?hours=N → グラフデータ（SWR, refreshInterval: 300_000 = 5分）
- GET /api/transactions      → 取引履歴（SWR, refreshInterval: 300_000）
```

> **自動更新:** スクレイパーが毎時実行のため、5分間隔のポーリングで十分。ユーザーがタブに戻った際（`revalidateOnFocus`）にも再取得する。

## 6. データ収集（GitHub Actions）

### 6.1 スクレイパー構成

```
.github/
  workflows/
    scrape.yml          # cron設定・実行
  scripts/
    scrape.py           # スクレイパー本体（並列実行・部分失敗許容）
    sites/
      ama_gift.py       # 各サイト固有のパーサー
      giftissue.py
      beterugift.py     # JSON対応の可能性あり
      amaten.py
    post_to_worker.py   # Workers APIへの送信
```

### 6.2 スクレイパーフロー

```
1. 各サイトのパーサーを並列実行（ThreadPoolExecutor）
2. 各パーサーは独立して成功/失敗を返す（例外をキャッチして None を返す）
3. 成功したサイトの出品一覧から集計値を計算（Python側）
4. 取引履歴を取得（対応サイトのみ）
5. 成功したサイトのデータのみ POST /api/ingest に送信
   - 全サイト失敗の場合のみ exit code 1 で GitHub Actions を失敗させる
   - 一部失敗はログに記録するが exit code 0（部分成功として扱う）
```

**タイムゾーン処理:** 各パーサーは `traded_at` を UTC ISO 8601 に変換して返す。JST 表示のサイトは `-9時間` を適用する。

### 6.3 エラーハンドリング・リトライ

| 状況 | 対応 |
|------|------|
| HTTPリクエスト失敗 | 最大2回リトライ（1秒待機）、それでも失敗は当該サイトをスキップ |
| HTMLパース失敗 | 当該サイトをスキップ、エラーログを標準エラーに出力 |
| Workers APIへの送信失敗 | 最大3回リトライ（指数バックオフ）、失敗で exit code 1 |
| 全サイト失敗 | exit code 1 → GitHub Actions のJob失敗通知が飛ぶ |

### 6.4 GitHub Actions ワークフロー概要

```yaml
name: Scrape Gift Card Prices
on:
  schedule:
    - cron: '0 * * * *'   # 毎時0分（UTC）
  workflow_dispatch:        # 手動実行も可能
jobs:
  scrape:
    runs-on: ubuntu-latest
    timeout-minutes: 10     # ハング防止
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
      - run: pip install requests beautifulsoup4 lxml
      - run: python .github/scripts/scrape.py
        env:
          INGEST_SECRET_TOKEN: ${{ secrets.INGEST_SECRET_TOKEN }}
          WORKER_URL: ${{ secrets.WORKER_URL }}
```

> **実行時間の目安:** pip install込みで1〜2分。パブリックリポジトリであれば GitHub Actions は無制限のため問題なし。

### 6.5 beterugift の特記事項

`https://beterugift.jp/home/load/3?_=タイムスタンプ` のURL形式はjQuery AJAXリクエストのパターンに一致する。スクレイピング実装前に `Content-Type: application/json` で直接GETし、JSONレスポンスが返るか確認する。JSON形式であればHTMLパースが不要となり実装が大幅に簡略化できる。

## 7. セキュリティ

| 脅威 | 対策 |
|------|------|
| POST /api/ingest への不正書き込み | Bearer トークン認証（GitHub Secrets + Workers 環境変数） |
| トークンの漏洩 | GitHub Secrets に保存、コードにはハードコーディングしない |
| 不正なデータのDB挿入 | Workers 側でホワイトリストバリデーション（§4.2参照） |
| 未来日時・古すぎる日時のデータ挿入 | `scraped_at` を現在時刻 ±1時間以内に制限 |
| スクレイピングのブロック | User-Agentの設定、サイトごとに適切なリクエスト間隔を設定 |
| D1への過剰書き込み | `UNIQUE` 制約 + `INSERT OR REPLACE` / `INSERT OR IGNORE` で重複を排除 |

## 8. Cloudflare 無料枠との整合性

| サービス | 無料枠 | 想定使用量 | 余裕 |
|---------|--------|-----------|------|
| Workers リクエスト | 100,000回/日 | ≒ 150回/日（24 ingest + 読み取り） | ✅ 十分 |
| Workers CPU時間 | 10ms/リクエスト | ingestは軽量なDB書き込みのみ | ✅ 問題なし |
| D1 書き込み | 100,000行/日 | ≒ 4行/時 × 24h = 96行/日 | ✅ 十分 |
| D1 読み取り | 5,000,000行/日 | フロントエンドアクセス次第 | ✅ 余裕あり |
| D1 ストレージ | 5GB | 1年=約35,000行（数MB） | ✅ 問題なし |
| Pages ビルド | 500回/月 | 開発時のみ | ✅ 十分 |
| GitHub Actions | **パブリックリポジトリなら無制限** | 1回/h × ≒1.5分 × 24h × 30日 = 1,080分 | ✅ 問題なし |

> **GitHub Actions の前提:** **リポジトリはパブリックにする**。プライベートの場合は月2,000分の制限があり毎時スクレイプでは超過する。スクレイパーのコードは公開されるが、`INGEST_SECRET_TOKEN` は GitHub Secrets で保護されるため実害はない。
